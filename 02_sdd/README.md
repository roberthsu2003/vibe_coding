# 2. SDD (軟體設計說明書 / System Design Document)

這對應到我們剛才討論的 `plan.md`。在傳統開發中，SDD 往往是大部頭的文件；但在 Vibe Coding 中，它是**「給 AI 的上下文（Context）」**。

- **教學重點**：如何將模糊的需求（`requirement.md`）轉化為 AI 讀得懂的技術規格（`plan.md`）。包括資料結構（Schema）、API 設計、檔案結構。
- **為何需要**：若沒有 SDD，AI 容易在迭代多次後「迷路」或產生「幻覺」（Hallucinations），導致前後程式碼風格不一致。
- **Vibe 觀點**：SDD 是為了減少「來回修復」的時間。它是 AI 的導航地圖。

---

## SDD的教學流程

步驟 1：AI 產生「需求模板」
↓
步驟 2：學生填寫模板（手寫粗略需求）
↓
步驟 3：學生把填好的模板丟給 AI，請 AI 產生 task.md
↓
步驟 4：學生審核 task.md（拆解合理嗎？順序對嗎？有漏掉嗎？）
↓
步驟 5：逐步執行任務，每完成一個就審核程式碼

## 這樣設計的好處

| **步驟**       | **學生學到什麼**        |
| ------------ | ----------------- |
| 填寫模板         | 學會把需求「講清楚」        |
| 審核 task.md   | 學會判斷「AI 的拆解邏輯對不對」 |
| 逐步執行 + 審核程式碼 | 學會「理解程式碼在做什麼」     |

**設計「需求模板」的具體格式**（讓學生好填、AI 好讀）

## 如何讓AI產生一個好的需求模板

學生想做某個功能（例如：待辦清單）
↓
學生上網找 2-3 個類似的 App 或網站
↓
學生觀察並選一個當參考
↓
把參考資料丟給 AI，請 AI 產生需求模板
↓
學生根據模板填寫自己的需求

## 讓AI產生一個好的需求模板的prompt

```javascript
我是程式初學者，正在學習如何寫需求規格。

附件是我從網路上找到的一個參考範例的截圖。

請你幫我：

1. 觀察這個畫面，列出你看到的「功能」有哪些
2. 針對每一個功能，產生一份需求模板

```

## 找參考範例的管道

| **管道**        | **適合找什麼**     | **範例網站**                         |
| ------------- | ------------- | -------------------------------- |
| **Google 搜尋** | 真實在運作的 App/網站 | 搜尋「todo app」「記帳 app」直接看產品        |
| **設計模板網站**    | UI 畫面、設計稿     | Dribbble、Behance、Figma Community |
| **UI 範例庫**    | 特定元件、功能的設計參考  | Mobbin、Screenlane、Page Flows     |

---

## 針對初學者，我的建議

| **類型**  | **推薦**        | **原因**                              |
| ------- | ------------- | ----------------------------------- |
| 最簡單     | **Mobbin**    | 依功能分類（登入、購物車、設定頁...），學生很容易找到特定功能的截圖 |
| 次簡單     | **Dribbble**  | 搜尋關鍵字就有大量 UI 設計圖，但品質參差不齊            |
| 直接看實際產品 | **Google 搜尋** | 最真實，但學生需要自己判斷哪些功能值得參考               |

## 關於計畫(plan.md)和任務(task.md)

## 先定義一下這兩個東西

| **文件**          | **內容**            | **粒度**    |
| --------------- | ----------------- | --------- |
| **plan.md（計畫）** | 功能的整體架構、模組劃分、先後順序 | 粗，偏「要做什麼」 |
| **task.md（任務）** | 具體的執行步驟，一步一步的指令   | 細，偏「怎麼做」  |

---

## 用蓋房子來比喻

|             | **比喻**                           |
| ----------- | -------------------------------- |
| **需求**      | 「我要一棟三房兩廳的房子，要有陽台」               |
| **plan.md** | 「先打地基 → 再蓋結構 → 再做水電 → 最後裝潢」      |
| **task.md** | 「步驟 1：挖地 50cm、步驟 2：灌水泥、步驟 3：...」 |

---

## 分析：需要 plan.md 嗎？

| **情況**                    | **建議**                   |
| ------------------------- | ------------------------ |
| **小專案**（功能少，例如只有 3-5 個功能） | 可以跳過 plan.md，直接產 task.md |
| **中大型專案**（功能多，有相依性）       | 需要 plan.md，先確認架構再展開任務    |
| **教學目的**                  | **建議保留 plan.md**，讓學生理解「為何這樣拆解」的思維過程，而非只給他們執行清單 |

---

## 教學目的的詳細說明

### 為什麼教學時建議保留 plan.md？

即使專案很小，在教學情境下仍建議讓學生練習撰寫 plan.md，原因如下：

| **學習目標** | **透過 plan.md 學到什麼** | **只用 task.md 的缺點** |
| -------- | ------------------- | ----------------- |
| **架構思維** | 學會先想「整體」再想「細節」      | 容易陷入「見樹不見林」，只顧執行任務  |
| **模組拆解** | 理解如何將大功能拆成小模組       | 不知道為何要這樣拆，只會照做      |
| **優先順序** | 學會判斷哪些功能應該先做        | 可能做錯順序，導致返工         |
| **溝通能力** | 練習用結構化方式表達想法        | 缺乏向 AI 或團隊清楚說明架構的能力 |

### 教學流程建議

**步驟 1**: 讓學生先寫 plan.md（粗略架構）
```markdown
# 計畫：待辦清單 App

## 核心功能模組
1. 任務管理（新增、刪除、標記完成）
2. 資料儲存（LocalStorage）
3. UI 介面（任務列表、輸入框、按鈕）

## 實作順序
1. 先做最基本的新增功能
2. 再加入刪除與標記完成
3. 最後加入資料持久化
```

**步驟 2**: 請 AI 根據 plan.md 生成 task.md
```markdown
請根據上面的 plan.md，幫我生成詳細的 task.md，
包含每個步驟的具體實作內容。
```

**步驟 3**: 學生審核 task.md
- 檢查拆解是否合理
- 確認順序是否正確
- 判斷有無遺漏

**步驟 4**: 逐步執行 task.md 中的任務

### 不同程度學生的調整

| **學生程度** | **plan.md 要求** | **AI 輔助程度** |
| -------- | ------------- | ----------- |
| **初學者**  | 只需寫出「要做哪些功能」  | AI 幫忙補充技術細節 |
| **中階**   | 需寫出「模組劃分 + 順序」 | AI 協助驗證合理性  |
| **進階**   | 需考慮「資料流 + 錯誤處理」 | AI 提供替代方案比較 |
