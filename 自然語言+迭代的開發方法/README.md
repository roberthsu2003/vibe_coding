# 自然語言 ＋ 迭代的開發方法

在 Vibe Coding 裡，你不用先寫規格書，也不用一次想清楚所有細節。只要**用自然語言說出目標**（例如：「開發貪食蛇遊戲」），讓 AI 產出**第一個原型**，再透過**迭代**一步步把應用程式做完善。

本章說明這套「先有一版，再慢慢修」的開發方式。

## 你會得到什麼（新手版）

- 一個「從一句話到可跑原型」的固定流程，不用先學完整語法
- 一組可直接複製的 Prompt 範本：開場、迭代、Bug 回報、驗收
- 一張「每一輪要檢查什麼」的清單，讓你知道下一句怎麼說
- 一個「避免越改越亂」的保存策略：小步改、每輪有記錄、可回退

## 快速開始（照著做就會動）

1. 先用「初始需求 Prompt」請 AI 產出第一版（能跑最重要）
2. 跑起來玩 30 秒，找出「目前的問題」或「你想加的下一個小功能」
3. 用「一次只改一件事」的迭代 Prompt，讓 AI 只做單一改動
4. 用「驗收清單」確認真的有改到你要的；不對就用「Bug 回報格式」補充
5. 每完成一輪就記錄（建議：Git commit），確保你隨時能回到上一版

> 建議搭配閱讀：[Prompt Engineering](../prompt/README.md)

## 1. 開發流程：一句話 → 原型 → 迭代完善

### 第一步：用自然語言說出目標

不需要寫技術規格，用平常說話的方式描述你想做的東西就好。

**例如：**
> 「請用 HTML 和 JavaScript 幫我寫一個網頁版的貪食蛇遊戲，可以用鍵盤控制蛇的方向，吃到東西會變長。」

### 第二步：AI 產出第一個原型

AI 會根據你的描述，產出一版**可以跑、可以玩**的程式。這一版通常已經具備基本功能：畫布、蛇會動、有食物、吃到會變長。

這一版不必完美，重點是**先有東西可以看、可以操作**。

### 第三步：用迭代讓應用程式更完善

拿到第一版之後，不要一次要求「全部重寫」，而是**一次改一點**，用對話引導 AI 補上你想要的東西。

**例如後續可以這樣迭代：**
- 「撞到牆或撞到自己要 Game Over，並顯示分數。」
- 「畫面上方要顯示目前分數。」
- 「Game Over 後要有一個『再玩一次』按鈕。」
- 「用 localStorage 紀錄最高分並顯示在畫面上。」

每完成一輪「描述 → AI 修改 → 你檢視」，應用程式就離你心中的樣子更近一步。這就是**迭代**。

---

## 2. 迭代的核心循環：Prompt → 生成 → 檢視 → 優化

迭代不是亂改，而是有節奏地推進。可以記住這個循環：

| 步驟 | 你做什麼 |
|------|----------|
| **Prompt（下指令）** | 用自然語言說出「目前還缺什麼」或「哪裡要改」 |
| **Generate（生成）** | 讓 AI 根據你的描述修改程式 |
| **Review（檢視）** | 實際跑一次，看結果是否符合預期 |
| **Refine（優化）** | 若有落差，再補一句更精準的指令，引導 AI 修正 |

> 💡 **心法**：接受第一版不完美，用「連續對話」逐步趨向理想結果。一次只專注一個小目標，改完再往下一個。

### 驗收清單（每一輪都用得上）

- 功能：你要求的那一項功能是否真的可用？（不是只有「看起來像」）
- 邊界：最常壞的情境有沒有測？（例如撞牆、連續按鍵、重新開始）
- UI：字有沒有被遮住？按鈕能不能點？有沒有提示訊息？
- 錯誤：有沒有 console error？有沒有卡死或無回應？
- 回歸：原本能用的功能有沒有被這次修改弄壞？

## 3. 為什麼「迭代」很適合新手？

- ⚡ **馬上看到成果**：不用先學完整語法，第一句指令就能得到可執行的原型，容易建立信心。
- 🧠 **不用一次想完**：不必在腦中規劃整個系統，可以「先要一個貪食蛇」→ 再「加計分」→ 再「加 Game Over 畫面」。
- 🛡️ **錯了再改就好**：AI 產出的程式若有問題，用自然語言描述哪裡不對、你希望怎樣，就能引導它修正。

### ⚠️ 避免「盲目重來」

不要只說「重寫一次」或「還是不對」而不說明**哪裡**不對、**希望變成怎樣**。迭代要有效，關鍵在**精準描述現狀與期望**，AI 才能對症下藥。

### 常見失敗模式（以及你可以怎麼說）

- 只說「不對」：改成「目前看到什麼 + 我期望什麼 + 重現步驟」
- 一次要求太多：改成「這一輪只改一件事：____」
- 叫 AI 大改架構：改成「先維持原本結構，先把功能補上／把 bug 修掉」
- 沒給驗收：改成「完成後我會用這些方式驗收：____」

## 4. 實戰示範：從「貪食蛇」第一版到完善版

下面用貪食蛇當例子，示範「自然語言開頭 → 第一版原型 → 迭代完善」的實際節奏。

### 第 0 輪：自然語言開場

你只說一句：
> 「請用 HTML + JavaScript 做一個網頁版貪食蛇，鍵盤控制方向，吃到東西蛇會變長。」

### 第 1 版原型（AI 產出後）

你得到一個可以玩的頁面：蛇會動、有食物、吃到會變長。但可能還沒有「撞牆死掉」「顯示分數」「再玩一次」。

### 第 2 輪迭代

你說：
> 「撞到邊界或撞到自己身體要 Game Over，並在畫面上顯示『Game Over』和目前分數。」

AI 修改後，遊戲具備結束條件與分數顯示。

### 第 3 輪迭代

你說：
> 「Game Over 之後要有一個『再玩一次』按鈕，點下去重新開始遊戲。」

### 第 4 輪迭代

你說：
> 「用 localStorage 紀錄最高分，畫面上方除了目前分數，也要顯示『最高分：OO』。」

就這樣，**從一句「開發貪食蛇遊戲」開始，不用寫規格書，靠自然語言與迭代，就能把應用程式從原型做到完善**。

## 5. 可直接複製的 Prompt 範本（新手最常用）

### A) 初始需求（先做出能跑的第一版）

把下面整段貼給 AI，再把「你想做的東西」那句改成你的題目：

```text
請用（你指定的技術：例如 HTML + JavaScript / Python / React）幫我做一個（你要做的東西）。

需求：
1) 最小可用：先做出「能跑、能操作」的第一版
2) UI 要簡單清楚，必要時加上使用說明
3) 請提供完整可執行的程式碼，並告訴我如何執行（例如直接開啟 HTML、或要怎麼跑指令）

完成後請附上：
- 你做了哪些功能（條列）
- 我可以怎麼驗收（條列）
```

### B) 迭代一次只改一件事（最重要的句型）

```text
我已經有一個可以跑的版本。請「只做以下一項改動」，不要順便重構或改其他地方：

目標（只做這一項）：
- （例如：撞到牆或撞到自己要 Game Over，並顯示分數）

限制：
- 盡量保留原本的程式結構與寫法
- 若需要新增變數/函式，請用清楚的命名

完成後請回覆：
- 你改了哪些檔案 / 哪些段落（條列）
- 我如何驗收它真的有成功（條列）
```

### C) Bug 回報格式（AI 才能精準修）

```text
這個功能有 bug，請你修正，並說明原因與修法。

目前現象：
- （例如：撞到牆不會 Game Over）

我期望的結果：
- （例如：撞到牆立刻停止、顯示 Game Over 與分數、禁止再移動）

重現步驟：
1) （例如：按方向鍵讓蛇往右一直走）
2) （例如：撞到右邊界）

我用什麼環境跑：
- 作業系統：
- 瀏覽器/版本（若是網頁）：
- 執行方式（例如直接開 HTML / 用 live server / 用 npm run dev）：

驗收方式：
- （例如：重現步驟做一遍，必須看到 Game Over；console 沒有 error）
```

### D) 想要 AI「先問我再改」避免失控（可選）

```text
在你動手改之前，請先用條列列出：
1) 你打算怎麼改（最多 5 點）
2) 你需要我補充的資訊（若有）
等我確認後你再開始修改程式。
```

## 6. 進階思考：要不要一開始就給完整需求文件？

有些人會把全部功能寫成一份 `Requirement.md` 一次丟給 AI。這樣做雖然快，但要注意：

- **門檻較高**：若對開發邏輯不熟，很難一次寫出嚴謹且不衝突的需求。
- **除錯成本大**：AI 若一次產出很多檔案或幾百行程式，出錯時很難定位問題。

> 🎯 **建議**：初期用「自然語言一句話 + 迭代」把應用程式做出來；等熟悉節奏後，再考慮用需求文件加速。

---

## 7. 迭代保存點（避免越改越亂）

- 每一輪只做一件事：加一個功能／修一個 bug／調一個 UI
- 每一輪都留記錄：至少寫下「這輪改了什麼 + 如何驗收」
- 建議用 Git：每輪成功就 commit，一旦改壞能立刻回退

```bash
git status
git add -A
git commit -m "迭代：加入 Game Over 與分數顯示"
```

## 總結

| 階段 | 做法 |
|------|------|
| **起頭** | 用自然語言說出目標（例如：「開發貪食蛇遊戲」） |
| **第一版** | 讓 AI 產出可執行的原型，不要求完美 |
| **完善** | 用迭代：每次描述一個「要加什麼 / 要改什麼」→ AI 修改 → 你檢視 → 再下一輪 |

先有一版能跑的，再用對話一步步完善——這就是自然語言 ＋ 迭代的開發方法。
