# 從一個 Prompt 到 PRD 與 Todo List

學會只需撰寫**一份 Prompt**，讓 AI 自動產生 `PRD.md` 與 `todolist.md`，並依據 PRD 內容推動 **todo list 的開發循環**。每完成一個步驟就**記錄並暫停**，讓使用者先測試，有助於在開發過程中即時矯正程式。

---

## 1. 流程總覽：一份 Prompt 驅動整個專案

```
你寫一份 Prompt（描述想做什麼）
        ↓
AI 產出 PRD.md（產品需求）＋ todolist.md（開發步驟清單）
        ↓
你依 todolist 一步一步開發：每步做完 → 記錄 → 暫停測試 → 再下一步
```

- **PRD.md**：定義產品要做什麼、給誰用、功能與規格，是開發的「北極星」。
- **todolist.md**：把 PRD 拆成可執行的步驟，用來推動開發循環。

---

## 2. 如何請 AI 產出 PRD 與 Todo List？

用自然語言描述你的專案目標，並明確請 AI 產出兩份檔案。

### 範例 Prompt

> 我想做一個網頁版的貪食蛇遊戲，用 HTML + CSS + JavaScript，單一 HTML 檔即可執行。  
> 請幫我產生兩份文件：  
> 1. **PRD.md**：產品需求文件，包含專案目標、目標使用者、功能清單、技術規格與驗收方式。  
> 2. **todolist.md**：開發待辦清單，把功能拆成「一步一步可實作」的項目，每個項目要具體、可測試，並標註依賴關係（例如先有畫布再有點擊事件）。

把這段（或依你專案改寫）貼給 AI，即可得到 `PRD.md` 與 `todolist.md` 的初稿，再依需要微調。

---

## 3. Todo List 的開發循環：做完一步就記錄、就暫停

Todo list 不是「一次全部做完」，而是**每完成一個步驟就記錄、就停下來**，讓使用者先跑一次、測一次，再進行下一步。這樣做的好處是：

- **即時矯正**：若這一步的程式有問題，可以馬上發現並修正，不會帶著錯誤往下堆。
- **可追溯**：每一步都有「完成／未完成」的記錄，專案進度與狀態一目了然。
- **降低風險**：小步前進，每次只改一點，除錯範圍小、心裡負擔也小。

### 建議的單步循環

| 步驟 | 你做什麼 |
|------|----------|
| 1. **做** | 依 todolist 選定「下一個待辦」，請 AI 產出或修改對應的程式。 |
| 2. **記錄** | 在 todolist 中把該項標記為已完成（例如打勾或寫上完成日期）。 |
| 3. **暫停** | **不要立刻進行下一項**，先停下來。 |
| 4. **測試** | 實際執行應用程式，操作剛完成的功能，確認是否符合 PRD 與預期。 |
| 5. **矯正** | 若有錯誤或落差，立刻用自然語言描述問題，請 AI 修正；修正後再測一次。 |
| 6. **下一步** | 確認沒問題後，再從 todolist 取下一個待辦，重複上述循環。 |

> 💡 **重點**：todolist 的規劃必須「做完一步就記錄、就停止」，讓使用者有機會測試。這樣開發過程中的程式矯正才會容易、有效。

---

## 4. PRD 與 Todo List 如何配合？

- **PRD.md** 回答「要做什麼、為什麼」：目標、使用者、功能、規格、驗收條件。開發過程中若方向偏了，可以回來看 PRD 對齊。
- **todolist.md** 回答「先做什麼、後做什麼」：把 PRD 的功能拆成可執行、可測試的小步驟，並在**每步完成後記錄與暫停**，推動「開發 → 測試 → 矯正」的循環。

兩份文件都由同一份 Prompt 驅動產出，之後再依實際開發與測試結果微調 PRD 或 todolist。

---

## 5. 技術說明（本教學範例）

- **技術棧**：HTML + CSS + JavaScript  
- **環境需求**：不需安裝任何環境或套件  
- **執行方式**：直接在瀏覽器中開啟 HTML 檔案即可運行  
- **資料儲存**：使用瀏覽器 `localStorage` 儲存遊戲記錄（若需求有提到）

每個遊戲的 README 內有對應的 PRD 範例與說明，可完全套用或依需求改寫後，再請 AI 產出你的 `PRD.md` 與 `todolist.md`，並依「做完一步 → 記錄 → 暫停測試 → 矯正 → 下一步」的循環開發。
